# batch10 認識改善提案書

作成日: 2026-04-01
対象: batch10 の vis.jpg で確認された誤認識の改善
優先方針: 認識速度より正確性を優先
前提: class 名は必ずしも物体の視覚的特徴を直接表さない

---

## 1. 概要

batch10 の 10 枚の確認結果は以下の通りです。

- 完全に正しい: 4 枚
- 誤認識あり: 6 枚

確認された誤認識パターンは次の 5 種です。

- **A**: 1 つの Instance を 2 つの Instance に誤分割する
- **B**: 2 つ隣接する Instance を 1 つに誤結合する
- **C**: 物体の一部が遮蔽され認識できない
- **D**: 似ていない物体でも、色が同じだけで誤認識する
- **E**: 参照画像不足により認識できない

本書では、これらの誤りを減らすための技術方針、実装案、参照画像ガイドライン、ならびに DINOv2 / DINOv3 を用いた検証案を整理する。

---

## 2. 基本戦略

正確性を優先する場合、単一段の認識で最終決定するよりも、**多段推論 + 整合性検査 + 再判定** の構成が有効である。

### 推奨アーキテクチャ

1. **候補抽出段**
   - recall 優先で instance 候補を広めに抽出する
   - 閾値はやや低めに設定し、見落としを減らす

2. **候補精査段**
   - 全体画像
   - 局所 crop
   - 周辺文脈込み crop
   - 高解像度 crop
   を用いて各候補を再判定する

3. **整合性補正段**
   - split / merge の補正
   - 境界の自然性確認
   - class ごとのサイズ・位置・共起関係の確認
   - 不確実候補の再推論

### この方針が有効な理由

- A/B は、境界や隣接関係の再評価で改善しやすい
- C は、文脈付き再判定や遮蔽学習で改善しやすい
- D は、色以外の特徴を強化することで改善しやすい
- E は、参照画像ライブラリや retrieval 的再判定で改善しやすい

## 3. 誤認識パターン別の対策一覧

| パターン | 現象 | 主原因 | 有効な対策 | 優先度 |
|---|---|---|---|---|
| A | 1 物体を 2 instance に誤分割 | 局所特徴への過依存、部分誤認識 | 境界付き merge 判定、part-whole 判定、周辺込み crop 再判定 | 高 |
| B | 2 物体を 1 instance に誤結合 | 接触境界の弱さ、解像度不足 | boundary head、split 後処理、高解像度再推論、instance embedding | 高 |
| C | 遮蔽で認識不可 | 遮蔽耐性不足、文脈利用不足 | 遮蔽 augmentation、amodal 的推定、文脈込み判定、multi-scale 推論 | 高 |
| D | 色だけで誤認識 | 色特徴への過依存 | color jitter、grayscale 混在、形状・質感重視、同色負例学習 | 高 |
| E | 参照不足で認識不可 | データ不足、variation 不足 | hard example 追加、参照画像拡充、retrieval 補助、active learning | 高 |

## 4. パターン別の詳細方策

### 4.1 A: 1 つの Instance を 2 つに誤分割する

#### 現象
1 つの物体の一部を別の物体として誤認し、複数 instance に分かれてしまう。

#### 主な原因
- 部分領域だけを見ると独立物体に見える
- 物体全体の構造より局所特徴が強く効いている
- 教師データの instance 分割基準が揺れている

#### 対策
- 隣接 mask 同士の **merge 候補判定** を行う
- **part-whole 判定器** を設ける
- 周辺文脈を含めた crop で再判定する
- containment ratio や boundary 接触率で後処理する

#### 実装メモ
- 小 instance が大 instance に大きく含まれる場合は再評価する
- class 一致かつ境界が弱い場合は統合候補とする

---

### 4.2 B: 2 つ隣接する Instance を 1 つに誤結合する

#### 現象
隣接・接触した 2 物体が 1 つの instance として出力される。

#### 主な原因
- 接触部の境界が曖昧
- 解像度不足により境界線が消える
- 1 mask にまとめる方が loss 上有利になっている

#### 対策
- **boundary / edge 予測ヘッド** を追加する
- watershed / distance transform 系の分離後処理を入れる
- 高解像度 crop で接触部を再推論する
- mask の凹み、くびれ、複数中心を split 兆候として扱う

#### 実装メモ
- 候補を 1 個に絞り切らず、複数分割仮説を保持する
- crowded scene 専用の hard example を増やす

---

### 4.3 C: 遮蔽で認識できない

#### 現象
一部しか見えていないため、モデルが同一物体として認識できない。

#### 主な原因
- 遮蔽下の学習データ不足
- visible 部分だけで判定しようとしている
- 周辺文脈を十分使えていない

#### 対策
- Cutout, Copy-Paste occlusion などで **遮蔽 augmentation** を強化する
- **amodal 的発想** を導入し、隠れた全体形状を推定する
- 周辺物体との関係や配置を使って再判定する
- global view と local view を併用する

#### 実装メモ
- 「一部だけでも同一と判断すべき事例」を明示的に教師化する
- 典型的な遮蔽パターンごとにデータを用意する

---

### 4.4 D: 色が同じだけで誤認識する

#### 現象
形状も用途も異なるのに、同系色という理由で誤判定する。

#### 主な原因
- 色特徴への過依存
- 形状・境界・質感の学習不足
- 負例設計が不十分

#### 対策
- **強めの color jitter** を導入する
- 一部 batch を **grayscale 化** して学習する
- 形状・境界・質感を重視する補助 loss を検討する
- 同色異物体の **negative pair** を学習させる

#### 実装メモ
- 色は完全に捨てず、補助的な証拠に格下げする
- 色一致のみで class を確定しない判定ルールにする

---

### 4.5 E: 参照画像不足により認識できない

#### 現象
人間なら過去知識や想像で補えるが、モデルは見たことがない variation に弱い。

#### 主な原因
- データ枚数不足
- variation の偏り
- rare case の教師不足

#### 対策
- 誤認識した画像を **hard example** として再学習に入れる
- rare / occluded / crowded / unusual context の参照画像を増やす
- retrieval ベースで近傍参照画像を使って再判定する
- active learning で迷った例を優先アノテーションする

#### 実装メモ
- 単純な枚数追加よりも、失敗要因ごとの coverage を埋めることを優先する

## 6. 推奨推論パイプライン

### Stage 1: Candidate Generation
- recall 優先で instance 候補を広く抽出
- confidence threshold はやや低め
- over-seg / under-seg の両方を許容して候補保持

### Stage 2: Candidate Verification
各候補について以下を実施する。
- 原画像全体から見た整合性確認
- 対象中心の局所 crop で再判定
- 周辺文脈込み crop で再判定
- 高解像度 crop による境界確認
- 必要に応じて別ヘッド・別モデルで補助判定

### Stage 3: Consistency Correction
- merge 候補判定
- split 候補判定
- mask containment check
- class-size-position の妥当性確認
- シーン内共起ルールの確認

### Stage 4: Escalation
- 確信度が低い instance のみ追加推論
- retrieval 参照や別 backbone で再チェック

## 7. 参照画像ガイドライン

参照画像は「枚数が多いほどよい」ではなく、**失敗しやすい variation を過不足なく含むこと** が重要である。

### 7.1 参照画像の役割

参照画像は以下の役割を持つ。

- class の典型例を学習させる
- 典型でない variation を補う
- 遮蔽・隣接・背景混同などの hard case を学習させる
- 誤認しやすい負例との区別を学習させる

### 7.2 参照画像に必ず含めたい variation

| 観点 | 具体例 | 必要理由 |
|---|---|---|
| 視点 | 正面、斜め、側面、俯瞰 | 視点依存を減らす |
| 距離・スケール | 近距離、中距離、遠距離 | サイズ変化への対応 |
| 回転 | 向き違い、傾き | 姿勢変化への対応 |
| 照明 | 明所、暗所、逆光、色温度差 | 色・コントラスト依存の低減 |
| 背景 | 単純背景、雑然背景、同系色背景 | 背景依存の低減 |
| 遮蔽 | 一部欠け、前景遮り、重なり | C 対策 |
| 隣接 | 同種隣接、異種隣接、接触状態 | B 対策 |
| 部分見え | 端切れ、半分のみ、角だけ | C/E 対策 |
| 似た色の負例 | 同色だが別物体 | D 対策 |
| 稀な文脈 | 通常と異なる配置・環境 | E 対策 |

### 7.3 推奨構成

各 class について、最低でも次の 4 層を意識して集める。

1. **典型例**
   - 最も代表的な見え方
2. **境界事例**
   - 隣接、接触、 crowded 状況
3. **遮蔽事例**
   - 一部しか見えない状況
4. **負例事例**
   - 誤認しやすい別物体

### 7.4 収集・管理ルール

- class ごとに variation タグを付ける
  - `occluded`
  - `crowded`
  - `same-color-negative`
  - `rare-view`
  - `small-object`
  - `unusual-context`
- 失敗画像は必ず保存し、再学習候補として管理する
- 参照画像追加時は、どの失敗タイプを補う画像かを明示する
- 同じ条件の画像ばかり増やさない

### 7.5 NG パターン

- 正面・中央・高画質の典型例ばかり集める
- 色違いが少ない
- 遮蔽例がほとんどない
- 隣接・重なり例がない
- 負例が未整備で「何と間違えるか」が設計されていない

### 7.6 参照画像レビュー項目

以下を class 単位で確認する。

- 視点 variation は十分か
- 遮蔽例はあるか
- 隣接例はあるか
- 同色負例はあるか
- 稀な文脈が含まれているか
- 直近の失敗例を反映したか

## 8. class 名が視覚特徴を直接表さない場合の方針

本件では、class 名が必ずしも見た目をそのまま表していない。
そのため、class を単なるラベル名ではなく、**多面的な定義セット** として扱う必要がある。

### class ごとに整理すべき情報

| 項目 | 内容 |
|---|---|
| 見た目特徴 | 形状、質感、境界、典型色 |
| 部分構造 | どの部分が見えれば判別できるか |
| サイズ感 | 典型サイズ、比率 |
| 位置傾向 | 画面内・シーン内での出現位置 |
| 共起関係 | 一緒に出やすい物体 |
| 使用文脈 | どういう場面で出るか |
| 遮蔽時の見え方 | 一部欠けたときの特徴 |
| 誤認しやすい負例 | 色だけ似ている物体、形だけ似ている物体 |

この情報を、学習データ設計・後処理ルール・参照ライブラリ構築に反映する。

## 9. DINOv2 / DINOv3 の試験導入案

細部に強い backbone として DINOv2 や DINOv3 の活用は十分検討価値がある。
特に、局所構造・質感・自己教師あり特徴が効くため、A/B/D/E の改善可能性がある。

### 9.1 期待できる効果

| 項目 | 期待効果 |
|---|---|
| 細部表現 | 小領域や局所構造の識別向上 |
| 類似色誤認の低減 | 色以外の特徴表現が強くなる可能性 |
| 参照検索との相性 | retrieval embedding として使いやすい |
| 遮蔽時の頑健性 | 部分特徴からの判定改善が期待できる |

### 9.2 注意点

- DINO 系 backbone を入れるだけで A〜E が自動解決するわけではない
- A/B は境界後処理や split-merge 戦略も同時に必要
- C/E はデータ設計と参照画像整備が依然として重要
- 推論コストやメモリ消費は増える可能性がある

### 9.3 試験の進め方

#### 比較対象
- 現行 backbone
- DINOv2 backbone
- DINOv3 backbone（利用可能なら）

#### 比較条件
- 学習データは同一
- 後処理条件もできるだけ同一
- 評価セットには A〜E の hard case を十分含める

#### 評価指標

| 指標 | 内容 |
|---|---|
| 全体精度 | mAP, AP50, AP75 など |
| A 改善率 | over-seg の減少 |
| B 改善率 | under-seg の減少 |
| C 改善率 | 遮蔽下の recall |
| D 改善率 | 同色負例への誤判定率低下 |
| E 改善率 | rare / low-reference case の recall |
| 推論時間 | 参考値として計測 |

### 9.4 最小実験計画

1. 現行 backbone をベースラインとして固定
2. DINOv2 を backbone にしたモデルを 1 系統試す
3. 可能なら DINOv3 も同条件で試す
4. batch10 と hard case set で誤りタイプ別に比較する
5. 良ければ multi-scale + boundary 後処理と組み合わせる

### 9.5 推奨判断基準

以下の条件を満たすなら試験継続価値が高い。

- A/B/D のいずれかで明確な改善がある
- rare case の recall が改善する
- 計算コスト増が許容範囲内である
- 現行パイプラインに統合可能である

## 10. 推奨ロードマップ

### Phase 1: 失敗分析の固定化
- batch10 の 6 枚を A〜E タグで再整理
- 複合要因も記録
- class 別に失敗件数を集計

### Phase 2: 低コスト改善
- hard example 追加
- color bias 抑制 augmentation
- context crop 再判定
- boundary ベース後処理

### Phase 3: 参照画像整備
- class ごとに variation チェックシート作成
- 欠落 variation を補充
- 同色負例・遮蔽例・隣接例を重点追加

### Phase 4: backbone 評価
- DINOv2 / DINOv3 を比較試験
- hard case 指標で比較
- 必要なら retrieval 併用も試す

### Phase 5: 高度化
- retrieval-augmented recognition
- amodal 補完
- graph-based consistency optimization

---

## 11. 最終提言

最も重要なのは、**誤認識の種類がすでに明確化されていること自体が大きな強み** である点である。
本件では、単にモデルを重くするより、次の順で改善する方が効果的である。

1. 失敗例を hard example として活用する
2. multi-scale / context crop / boundary 後処理を導入する
3. 参照画像を variation ベースで再整備する
4. 色依存を下げる学習を行う
5. DINOv2 / DINOv3 を比較評価する

特に、**参照画像の質と coverage** は E だけでなく C/D にも効くため、優先度が高い。
また、DINO 系 backbone は試す価値が高いが、後処理やデータ設計を組み合わせて初めて真価が出る可能性が高い。
